# 新增信用卡方法論

這份流程用於之後複用。當有人說 `add 某張信用卡`、`新增 某卡`、或補充公開卡片資料時，依此流程整理成可查核、可推薦、可更新的公開資料。個人持卡、額度與帳戶狀態不得寫入 tracked public files，分層規則見 [[00-規範/公開私有分層|公開私有分層]]。

## 目標

新增信用卡不是只建立卡片頁，而是把「來源、公開卡片規則、公開銀行規則、通路、觀察清單、索引與 log」一次串好。最後的資料要能回答：

- 這張卡目前是否已持有，或只是候選？
- 官方回饋有效期間到何時？
- 推薦時能估多少回饋？
- 需要哪些條件、登錄、App 方案、帳戶自扣或指定支付？
- 回饋上限怎麼算？
- 所屬銀行是否共用額度；公開公式是否會受信用額度影響？
- 哪些資料還不能直接拿來推薦？

## 輸入解析

先把使用者輸入拆成結構化資訊。

| 輸入 | 要記錄的資訊 | 處理方式 |
|---|---|---|
| `add 某卡` | 卡名、銀行、可能別名 | 查官方產品頁確認正式名稱與權益；未明講持有時，只代表收錄公開資料，不自動視為個人持有 |
| `我有某卡` / `新增某卡 額度：<金額>` | 個人持卡或額度狀態 | 不寫入 tracked public files；公開層只整理該卡官方規則 |
| `研究某卡` / `候選某卡` / `非持有` | 公開研究需求 | 公開層照常建卡片規則或研究 backlog；不記個人是否持有 |
| `額度：<金額>` | 銀行信用額度 | 不寫入公開卡片頁、公開銀行頁、來源頁、研究清單或 log |
| `原 gogo visa / 太陽卡 mastercard` | 原卡名、付款組織、是否更名或產品整併 | 公開卡片頁只記錄官方更名或產品整併，不記特定個人的持有版本 |
| `我的等級是 大戶` | 會影響回饋估算的個人狀態 | 不寫入公開資料；公開卡片/銀行頁只保留官方等級規則 |
| 額度未記錄 | 額度未記錄 | 明確標示 `未記錄`，不要猜測 |

## 查核來源

新增卡片必須查官方來源。若官方資料不足，再把第三方資料放進觀察清單，不可直接當推薦依據。

優先查：

- 銀行官方信用卡產品頁。
- 官方權益方案頁或活動頁。
- 官方 PDF 條款、注意事項、回饋計劃。
- 更名、停發、權益變更公告。
- App 公告或活動登錄頁，若官方網頁明確指向該頁。

查核時至少抓出：

- 查核日期。
- 有效期間。
- 回饋率與回饋形式，例如現金回饋、點數、刷卡金。
- 回饋上限與計算公式。
- 登錄、領券、App 切換、自扣、帳戶等級、電子帳單等條件。
- 支付型態限制，例如實體卡、Apple Pay、LINE Pay、台新Pay、全支付。
- 排除條款與不回饋交易。
- 需複查日期。

## 來源筆記

先建立來源筆記，再整理 wiki。

位置：

- 官方活動或產品權益：`10-來源/官方活動/YYYY-MM-DD-銀行-卡名回饋查核.md`
- 條款 PDF 或注意事項：`10-來源/條款/YYYY-MM-DD-銀行-條款名稱.md`

來源筆記必備區塊：

- frontmatter：`type`、`source_type`、`bank`、`card`、`captured_on`、`effective_from`、`effective_to`、`review_on`、`status`
- `來源`：官方連結或 PDF
- `查核結論`
- `目前可用回饋` 或 `權益分級`
- `使用條件`
- `回饋上限`
- `排除與限制`
- `疑點與後續複查`

## 卡片頁

建立或更新 `20-Wiki/信用卡/銀行-卡名.md`。

卡片頁必備內容：

- 定位：一句話說清楚這張卡的推薦角色。
- 目前可用回饋表：情境、回饋、上限、有效期間、來源。
- 使用條件：登錄、綁定、任務、帳戶要求。
- 公開銀行關係：只標示所屬銀行、是否共用額度這個公開規則與銀行頁連結，不寫使用者目前額度。
- 排除條款。
- 適合情境與不適合情境。
- 來源與更新紀錄。

若回饋取決於 App 方案或等級，不要只寫最高回饋，必須列出公開成立條件。個人條件是否已達成不得寫入公開資料。

## 銀行頁與額度

只要新增銀行層級公開規則，就建立或更新 `20-Wiki/銀行/銀行名稱.md`。持卡資料、信用額度、調額紀錄與使用者帳戶狀態不得寫入公開銀行頁。

公開銀行頁要維護：

- 共用額度制度、點數系統、任務制、帳戶綁定等官方規則。
- 會影響多張卡的公開限制，例如指定帳戶自扣、電子帳單、分期或大額消費規則。
- 官方來源。

若輸入含個人額度，只能用來提醒「公開資料不得保存此資訊」。公開層只記錄銀行是否共用額度、回饋公式是否與信用額度連動，以及官方來源。

## 個人條件處理

公開 repository 不保存個人條件。公開卡片頁存在，不代表任何人持有。

常見條件：

- 銀行帳戶等級，例如 DAWHO 大戶。
- 信用卡權益等級，例如 CUBE Level。
- App 當日方案，例如 CUBE 權益方案、Richart Life 方案。
- 自動扣繳、電子帳單、領券、登錄。
- 卡別或付款組織，例如 JCB、Visa、Mastercard。
- 信用額度衍生的指定消費上限。

寫法原則：

- 公開頁只寫官方條件本身。
- 不寫特定個人是否已達成條件。
- 若需要個人條件才能推薦，在公開觀察清單標為「個人化推薦需另行判斷」。

## 通路與策略

若同一通路有兩張以上卡片、或某張新卡會影響既有通路推薦，更新 `20-Wiki/通路/通路名.md`。

常見通路：

- LINE Pay
- 國外實體商店
- 悠遊卡自動加值
- 指定行動支付
- 網購、外送、百貨、保費、旅遊住宿

通路頁要分清楚：

- 通路本身。
- 付款工具。
- 點數載體。
- 是否需要指定 App 方案、領券、登錄。
- 是否排除第三方支付、分期、代收、便利商店、全聯等。

若公開資料仍不足以形成通用推薦，先放入 `觀察清單.md` 的候選通路。若只是個人條件未記錄，不要寫入公開觀察清單。

## 觀察清單

以下狀況必須更新 `觀察清單.md`：

- 有效期間即將結束，通常抓結束日前 2 週作為複查日期。
- 官方產品頁與活動細則數字不一致。
- 回饋需登錄、限量、領券或 App 狀態，且公開活動狀態需要複查。
- 卡片是否停發、改名、整併或卡別影響仍需複查。
- 需要後續拆通路頁或策略頁。

觀察清單分三類更新：

- `即將複查`
- `公開條款查核結論`
- `候選卡片` / `候選通路`

非公開個人狀態不納入 tracked public files。

## Index 與 Log

每次新增卡片後更新：

- `index.md`：把新卡、新銀行、新通路放入目前已建公開列表。
- `log/YYYY-MM.md`：記錄本次新增了什麼、查核了哪些官方來源、同步了哪些公開頁；不得重述個人額度與持卡狀態。

log 要能讓之後的人快速知道這次更動的範圍，不需要重寫所有條款。

## 決策矩陣

公開推薦矩陣或策略頁只放公開資料足以支撐的結論。以下情況不要寫進正式推薦：

- 回饋有效期間已過或即將到期但未複查。
- 官方來源不足。
- 必要個人條件未知，且未知會改變推薦排序。
- 只有產品頁 headline，缺少細則、上限或排除條款。

可以先把卡放在 `觀察清單.md` 的候選卡片，等條件確認後再進矩陣。

## 完成檢查

新增卡片完成前，至少檢查：

```sh
rg -n "銀行名|卡名|別名|主要通路" .
find 10-來源 20-Wiki 30-研究 log -type f | sort
rg -n "目前信用額度|使用者提供|本次提供|持有中($|[[:space:]，。；、])|額度查核日期|可用額度|帳單週期[:：]|繳款截止[:：]" 20-Wiki 10-來源 30-研究 觀察清單.md index.md log
```

人工檢查：

- 來源頁存在，且官方連結可追溯。
- 卡片頁有回饋、上限、有效期間、來源與排除條款。
- 公開銀行頁不含使用者額度或持卡狀態。
- 沒有用公開卡片頁存在與否推定任何人的持有狀態。
- 相關通路頁已更新，或已放入候選通路。
- 觀察清單有下一次複查日期。
- index 與當月 log 已更新。

## 輸出給使用者

回覆使用者時簡短列出：

- 已新增的卡片頁。
- 已同步的公開銀行規則與信用額度公式。
- 重要限制、公開複查項或非公開個人狀態。
- 官方來源連結。

若輸入含個人額度或持卡狀態，要明確說「這些資料未寫入公開 repository」。
