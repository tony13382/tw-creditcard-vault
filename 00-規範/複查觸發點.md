# 複查觸發點

這份 SOP 規定 `review_on` 怎麼定、哪些事件強制觸發複查、如何避免把 review 集中在同一天。

## 為什麼要管 review_on

如果所有卡都 `review_on: 2026-06-15`，到 6/15 就要一次處理 60+ 張卡 → 不可行。

設計原則：
1. **散點配置**：同一週內到期的卡不超過 5-10 張
2. **跟隨活動週期**：活動結束日 + 2 週為標準 review_on
3. **跟隨銀行節奏**：H1 / H2 銀行通常 1 月 / 7 月公告新辦法，往後 2 週 review
4. **永久條款用長 review**：終身免年費、無到期條款的卡，review 設 12 個月後

## 預設 review_on 規則

### 規則 1：跟隨活動期間

若回饋有明確有效期間，`review_on = 活動結束日 + 2 週`：

| 活動期間 | review_on |
|---|---|
| 2026-01-01 至 2026-06-30（H1）| 2026-07-15 |
| 2026-07-01 至 2026-12-31（H2）| 2027-01-15 |
| 2026 Q1（1-3 月）| 2026-04-15 |
| 2026 全年 | 2027-01-15 |
| 至 2026-12-31 | 2027-01-15 |

### 規則 2：永久條款

無到期日的常設條款（基本回饋、年費政策、權益手冊）：

| 卡片類型 | review_on |
|---|---|
| 標準現金回饋卡 | 6 個月後（半年複查官方頁） |
| 頂級卡 / 高階卡 | 12 個月後（變動率低）|
| 邀請制 / 停辦 | 12 個月後 |
| 別名指向頁 | 12 個月後 |

### 規則 3：新卡 / 新活動

| 情境 | review_on |
|---|---|
| 新卡上市（< 3 個月）| 上市後 3 個月（追蹤實際使用回饋）|
| 新戶活動（如 90 天禮）| 活動截止日 + 1 週 |
| 限額名次活動（如「前 1,000 名」）| 1 個月內查是否額滿 |

### 規則 4：散點避免集中

若同一日已有 ≥ 5 張卡的 `review_on`，新卡往前後 3-7 天微調。可用：

```
跑 Grep 在 20-Wiki/信用卡/ 找出 review_on: 2026-07-15 的卡片數
→ 若 ≥ 5 張 → 新卡改為 2026-07-08 / 2026-07-22 等附近日期
```

## 強制觸發複查的事件

下列事件**立即**觸發複查（不論 `review_on` 是否到期）：

### S1：銀行官方公告變動

| 事件 | 動作 | 時限 |
|---|---|---|
| 銀行宣布**停發**某卡 | 卡片 `status` 改為 `stopped_issuance_existing_only`，加入該銀行「停辦卡群索引」 | 24 小時內 |
| 銀行宣布**新卡上市** | 建 `10-來源/` 來源筆記 + 初版 wiki（可 skinny）| 7 天內 |
| **年費政策**變動 | 卡片頁 `annual_fee` 更新 + 更新紀錄 | 24 小時內 |
| **權益手冊**改版 | 排除條款 / 旅平保期 / 機接條件更新 | 7 天內 |
| **回饋條款**重大變動（含活動延期 / 取消）| 「目前可用回饋」表更新 + 更新紀錄 | 7 天內 |

### S2：產業 / 監管變動

| 事件 | 動作 | 時限 |
|---|---|---|
| 銀行**併購 / 業務轉移**（如花旗 → 星展）| 建「業務轉移」索引頁 + 既有卡換卡建議 | 7 天內 |
| 金管會**新公告**（如 2024-01-10 分期不享回饋）| 影響所有適用卡，加入跨銀行規則頁 | 14 天內 |
| 新發卡機構進入金管會清單 | `30-研究/發卡機構清單.md` 加入 | 該月內 |

### S3：日期邊界（自動觸發）

| 日期 | 動作 |
|---|---|
| **每年 1 月 1 日** | H1 開始 → 抓所有銀行 H1 活動辦法，更新對應卡片 |
| **每年 7 月 1 日** | H2 開始 → 同上 |
| **每年 6 月 25 日** | H1 將結束 → 跑「過期描述掃描」（見 [[每日掃描節奏]] § 自動化）|
| **每年 12 月 25 日** | H2 將結束 → 同上 |
| **每月 1 日**（建議） | 跑「本月到期 review_on」批次處理 |

### S4：内部觸發

| 事件 | 動作 | 時限 |
|---|---|---|
| 我準備推薦這張卡（給自己 / 他人）| 跑[[品質checklist]]，補完未通過項目 | 當下 |
| 發現重複建檔 | 合併為主檔 + 別名頁 | 當下 |
| 發現第三方數字與官方矛盾 | 重新引官方頁，於更新紀錄註記差異 | 當下 |

## review_on 更新動作

每次 review 完成後，必須：

1. 在 `## 更新紀錄` 加一行：
   ```
   - YYYY-MM-DD：YYYY H[1/2] 補核。[結果]。複查日期推遲至 YYYY-MM-DD。
   ```
2. 更新 YAML `review_on` 至下次合理日期（依規則 1-4）
3. 若條款有重大變動 → 同時更新「目前可用回饋」表 + 來源筆記
4. 若 H2 活動未公告 → review_on 推遲到「預期公告日 + 2 週」

## status 變遷流程

```
        skinny_pending → active_with_conditions
                            ↓
        (邀請制)         (活動取消)         (銀行併購)
            ↓                ↓                  ↓
        invitation_only    stopped_issuance    business_transferred
                            ↓
                        merged_into_*（若有合併）
```

每次 status 變遷必須：
- 更新 YAML
- 加入對應索引頁（停辦索引 / 業務轉移索引）
- 在更新紀錄寫明變遷原因 + 日期

## 不適用的事件（不必觸發複查）

下列**不需**觸發 review：

- 銀行行銷話術變動（如 logo、卡面、宣傳片）
- 卡名稱微調（除非影響檢索）
- 排版 / typo 修正
- 對應銀行的非信用卡產品變動（debit card、存款帳戶）
- 個人卡內權益重新計算（額度、信用評分），這類不入公開頁

## 對 vault 的散點目標

理想分布（按月）：

| 月份 | 預期 review_on 卡數 | 原因 |
|---|---:|---|
| 1 月 | 30-50 | H1 公告複查 |
| 4 月 | 10-20 | Q1 結束 |
| 7 月 | 30-50 | H2 公告複查 |
| 10 月 | 10-20 | Q3 結束 |
| 其他月 | 5-15 | 散點 |

**警訊**：某月超過 60 張，下次新卡或 review 時往前後分散。

## 自動化檢查

```
跑 Grep 在 20-Wiki/信用卡/ 找出所有 review_on:，
依日期分組統計，列出每週 review 卡數。
若某週 > 10 張 → 列出該週的卡片，建議散點。
```

```
列出本週 (今天 - 今天+7) 所有 review_on 到期的卡片：
- 銀行 / 卡名 / 當前 status / 上次更新日期
```

```
查所有 active_with_conditions 但 6 個月未更新的卡（更新紀錄最後一行 < 今天 - 180 天）：
標為「需要補核」並加入觀察清單。
```
