# tw-creditcard-vault

台灣信用卡公開資料整理庫。以 Obsidian vault 的形式維護一套**可查核、可更新、可推理**的信用卡知識庫，只收錄能由官方來源驗證的卡片權益、銀行規則、通路條件、活動條款與研究 backlog。

> 這個 repository 只放**公開資料**。任何個人持卡狀態、實際信用額度、申辦／核卡日期、帳單繳款紀錄、登錄／自扣狀態等個人資訊，都不會進入這個 repo。

## 為什麼做這個

市面上的信用卡比較資訊散在各銀行官網、活動頁、App 公告、部落格、論壇之間，且**回饋率、上限、排除條款、登錄要求隨時會變**。一張卡這個月划算，下個月加碼結束、上限降低、通路被剔除，推薦邏輯就完全不同。

這個 vault 用「來源優先、政策可追溯、有效期可查核」的方式整理，避免把過期或二手資訊當推薦依據。

## 範圍

收錄：

- 銀行與發卡機構的公開規則（共用額度、點數系統、任務制、官方帳戶條件）。
- 信用卡公開權益、回饋上限、有效期間、成立條件與排除條款。
- 通路定義、指定商店、付款工具限制與帳單辨識規則。
- 官方活動頁、條款 PDF、限時加碼活動的索引與查核筆記。
- 待研究的發卡機構與信用卡 backlog。

不收錄：

- 個人持卡清單、信用額度、可用額度、調額紀錄。
- 個人帳戶等級、會員等級、自扣、電子帳單、登錄、領券或 App 方案狀態。
- 申辦日、核卡日、帳單日、繳款日、消費紀錄、偏好或任何身分識別資料。

## 編輯原則

- **先保存來源，再整理 wiki**。每張卡至少要有一個官方來源筆記。
- **官方來源優先**：銀行官網、權益頁、活動頁、PDF 條款、App 公告。
- **非官方來源只當線索**，不能直接當推薦依據；需標示為二手資訊。
- 每筆回饋資訊都要記錄 `查核日期`、`有效期間`、`回饋上限`、`限制條件`、`排除條款`、`需複查日期`。
- 過期、未查核、來源不明的資料不能直接進入最終建議，只能放入 `觀察清單.md` 或來源筆記。
- 同一張卡有 2 個以上來源或會被多個策略引用時，才升級到 `20-Wiki/信用卡/`。

更完整的編輯規範見 [`CLAUDE.md`](./CLAUDE.md) 與 [`00-規範/`](./00-規範/)。

## 推薦輸出格式

從這個 vault 產出的卡片推薦，建議使用統一格式：

```
- 推薦卡片：
- 適用情境：
- 預估回饋：
- 回饋上限：
- 主要限制：
- 排除條款：
- 查核日期：
- 來源：
- 需複查日期：
```

公開輸出**不得**包含個人信用額度或持卡狀態。

## 發布檢查

每次推送前，跑下面的 ripgrep 命令掃描是否誤把私人資料納入：

```sh
rg -n "40-私有|目前信用額度|可用額度|持有中($|[[:space:]，。；、])|額度來源：使用者|使用者提供|本次提供|目前已記錄|核卡日期[:：]|帳單日[:：]|繳款截止[:：]|自動扣繳：未記錄|信用額度 (80,000|150,000|160,000|250,000)|額度 (80,000|150,000|160,000|250,000)" CLAUDE.md index.md log 10-來源 20-Wiki 30-研究 觀察清單.md
```

命中不一定代表錯誤，公開條款可能會提到「信用額度加計固定金額」或「需自動扣繳」這類字串。但只要命中個人狀態、實際額度或本機私有路徑（例如 `40-私有/`），就**不要推送**。

## 貢獻

歡迎以 issue 或 PR 補上：

- 新銀行或新卡的官方來源連結。
- 既有卡片的條款變動、活動更新、權益異動。
- 通路或策略頁的新案例。

PR 規則：

1. 任何卡片變動都要附**官方來源網址**與**擷取日期**。
2. 不接受純第三方部落格資料作為唯一來源。
3. 不要在 PR 內附上個人持卡資訊、信用額度或帳戶狀態。
4. 命名規則：來源 `YYYY-MM-DD-銀行-活動或條款名稱.md`，卡片 `銀行-卡名.md`。
5. 提交前請跑上面的發布檢查。

## 工具

這個 vault 設計成可在 [Obsidian](https://obsidian.md) 中以雙向連結瀏覽，但純 Markdown 在任何編輯器或 GitHub 上也可閱讀。`[[wiki-link]]` 語法在 Obsidian 中會自動連結，在 GitHub 上會以純文字顯示。
